# S4 Integrating Security into Development Strategy

## S4.3 The Role of Merge Reviews

The merge review (pull request review) serves as a crucial point for guidance and validation. For security, this is where **securable code review can scale effectively**. It acts as an agile training ground where developers learn from peers and receive feedback.

### Key Principles

| Principle | Description |
| :--- | :--- |
| Guardrails not gates | Use merge reviews as guardrails to grow the FIASSE mindset, not as blocking gates |
| SSEM attributes as basis | SSEM attributes provide a concrete basis for review criteria |
| Threat modeling at code level | Ask "What can go wrong?" to examine code-level issues threatening Trustworthiness and Reliability |
| Positive experience | Create a constructive environment for developers |

### Benefits

| Benefit | Evidence |
| :--- | :--- |
| Most effective early detection | Code review through merge requests is the single most effective technique for identifying security vulnerabilities early [OWASP-CRG] |
| Fewer post-release bugs | Teams implementing structured reviews report up to 80% fewer post-release bugs [CodeReviewBenefits] |
| Knowledge sharing | Collaborative reviews share insight and expertise, providing fresh perspectives |
| Context-aware | Human reviewers understand change context and system architecture, unlike automated scans |
| Skill elevation | FIASSE-trained AppSec professionals elevate overall team capability over time |
