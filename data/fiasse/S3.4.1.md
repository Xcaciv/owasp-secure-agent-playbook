# S3 The Securable Principle

## S3.4 Measuring SSEM Attributes

Measuring SSEM attributes is essential to quantify and evaluate the securable qualities of software. These measurements provide actionable insights to guide developers in enhancing securability.

### S3.4.1 Measuring Maintainability

#### S3.4.1.1 Analyzability

| Metric | Type | Description |
| :--- | :--- | :--- |
| Volume (LoC) | Quantitative | Tracked per module/system; lower LoC for given functionality indicates better analyzability |
| Duplication Percentage | Quantitative | Measured by static analysis (e.g., SonarQube, PMD); lower is better |
| Unit Size (mLoC/cLoC) | Quantitative | Average lines per method or class; excessively large units are harder to analyze |
| Unit Complexity | Quantitative | Cyclomatic complexity measured by static analysis; lower per unit is better |
| Comment Density/Quality | Quantitative | Ratio of comment lines to code lines, or qualitative review of comment usefulness |
| Time to Understand (TTU) | Qualitative | Average time for an unfamiliar developer to understand a code section's purpose and flow |
| Developer Surveys | Qualitative | Periodically ask developers to rate analyzability of modules they work with |

#### S3.4.1.2 Modifiability

| Metric | Type | Description |
| :--- | :--- | :--- |
| Module Coupling (Afferent/Efferent) | Quantitative | Incoming and outgoing dependencies; lower afferent coupling = easier to change |
| Change Impact Size | Quantitative | Files/modules typically affected by a common change type; smaller is better |
| Regression Rate | Quantitative | Percentage of changes introducing new defects; lower is better |
| Ease of Change Assessment | Qualitative | During code reviews, assess how easy a hypothetical related change would be |
| Time to Implement Change | Qualitative | Average time to implement standard modifications or enhancements |

#### S3.4.1.3 Testability

| Metric | Type | Description |
| :--- | :--- | :--- |
| Code Coverage | Quantitative | Percentage of code covered by automated tests; higher is generally better |
| Unit Test Density | Quantitative | Number of unit tests per KLoC or per class/module |
| Mocking/Stubbing Complexity | Quantitative | Difficulty or setup required to isolate units for testing |
| Ease of Writing Tests | Qualitative | Developer feedback on ease of writing meaningful tests |
| Test Execution Time | Qualitative | Excessively long test suite runs can disincentivize testing |
