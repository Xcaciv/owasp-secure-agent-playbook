# S3 The Securable Principle

## S3.2 Core Securable Attributes

### S3.2.2 Trustworthiness

**Definition**: "The degree to which a system (including all its components and procedures) can be expected to achieve a set of requirements, such as security requirements" (RFC 4949).

A trustworthy system operates within defined levels of trust and meets specified security properties. FIASSE focuses on the inherent qualities of the code that enable trustworthiness, requiring strong architectural design, clear trust boundaries, and flexible areas of the codebase.

---

#### S3.2.2.1 Confidentiality

**Definition**: "The property that data is not disclosed to system entities unless they have been authorized to know the data" (RFC 4949).

FIASSE emphasizes confidentiality as a fundamental attribute achieved by designing software with inherent qualities that protect sensitive information, rather than relying solely on overlaid security controls.

| Aspect | Security Implication |
| :--- | :--- |
| Data classification | Sensitive information must be identified and categorized |
| Authorization design | Access to data must be controlled based on authorization |
| Inherent protection | Software design itself should protect confidentiality |
| Beyond controls | Not solely dependent on overlaid security controls |

---

#### S3.2.2.2 Accountability

**Definition**: "The property of a system or system resource that ensures that the actions of a system entity may be traced uniquely to that entity" (RFC 4949).

This involves managing principals and access to enable the attribution of actions to specific users or processes, essential for auditing and incident response.

| Aspect | Security Implication |
| :--- | :--- |
| Unique attribution | Every system action verifiably linked to a specific entity |
| Comprehensive logging | Robust logging of all security-relevant actions |
| Secure authentication | Strong authentication supporting traceability |
| Non-repudiation | Leverages strategies common to non-repudiation |

---

#### S3.2.2.3 Authenticity

**Definition**: "The property that an entity is what it claims to be" (ISO/IEC 27000:2018).

Authenticity is strongly supported by non-repudiation: "the ability to prove the occurrence of a claimed event or action and its originating entities" (ISO/IEC 27000:2018).

| Strategy | Description |
| :--- | :--- |
| Secure authentication | Multi-factor authentication and strong credential types |
| Digital signatures | Verify origin and integrity of data and communications |
| Comprehensive logging | Trace actions back to their origin |
| Transparency | Complements accountability by ensuring tracing mechanisms are clear |
