# S6 Practical Guidance for Secure Software Development

## S6.2 Threat Modeling

Threat modeling is a valuable design activity. Start with a simple framework rather than discouraging those who would like to identify things that can go wrong.

### Four Question Framework

| Question | Purpose |
| :--- | :--- |
| What are we building? | Define the system scope and architecture |
| What can go wrong? | Identify potential threats and attack paths |
| What are we going to do about it? | Determine mitigations and controls |
| Did we do a good job? | Verify effectiveness of chosen mitigations |

### S6.2.1 Code Level Threat Identification

Asking "What can go wrong?" at the code level helps identify potential issues and develops junior engineers' sense of taste for good code.

| Context | Application |
| :--- | :--- |
| Merge Reviews | Scope review to changeset of a merge for clear context and responsibility |
| Static Analysis | Review results (bonus if scoped to merge request) with intent of going deeper using the Four Question Framework |
| Pair Programming | Facilitates knowledge sharing, prevents security issues, and builds comfort asking "What can go wrong?" |

Code-level threats should feed back into the Threat Model for assessment from a design perspective, potentially highlighting design solutions that scale better.

### S6.2.2 Threat Modeling Solution Framework

Threats follow data. Understanding data flows is crucial for identifying vulnerabilities. STRIDE gives structure to thinking about attacks; **SSEM can structure what we do about them**:

| Approach | Description |
| :--- | :--- |
| STRIDE | Framework for thinking about attack categories |
| SSEM attributes | Considering Trustworthiness and Reliability can lead to existing architectural or logical solutions |
| Design-level solutions | When a solution is not an attribute of the system, it must be addressed through defining or refining security requirements |
