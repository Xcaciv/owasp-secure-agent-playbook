# S3 The Securable Principle

The Securable Principle shifts from asking "Is it secure?" to a more nuanced approach: **"Do we meet our defined goals for this particular securable attribute?"** Security is not a binary state but a spectrum of qualities that can be measured, improved, and maintained.

## S3.1 Securable Software Engineering Model Overview

SSEM is centered on a design language that uses established software engineering terms to define core attributes that make software "securable." These attributes abstract security away from specialized jargon or exploit-centric views.

### SSEM Attribute Model

| **Maintainability** | **Trustworthiness** | **Reliability** |
|:---------------------|:---------------------:|----------------:|
| Analyzability        | Confidentiality       | Availability    |
| Modifiability        | Accountability        | Integrity       |
| Testability          | Authenticity          | Resilience      |

### Design Objectives

SSEM is designed to:

| Objective | Description |
| :--- | :--- |
| Iterative alignment | Account for the iterative nature of software development and agile methodologies |
| Flexible application | Serve as a mental model, a checklist, or a means to express and set clear expectations |
| Scalable process | Scale effectively without requiring overly complex processes that conflict with development workflows |
| Nuanced assessment | Shift from binary "secure/insecure" to "Do we meet our goals for attribute X?" |
| Engineering language | Use terms already familiar to software engineers |

SSEM emphasizes the inherent qualities of software that contribute directly to security, allowing it to scale without conflicting with development team workflows.
