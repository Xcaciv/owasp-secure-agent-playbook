# S6 Practical Guidance for Secure Software Development

## S6.4 Resilient Coding and System Resilience

Resilience refers to an application's ability to continue running predictably, even under unfavorable circumstances or load. Tactical resilience is achieved through **defensive coding** practices.

### Defensive Coding Practices

| Practice | Description |
| :--- | :--- |
| Strong typing | Ensure data is usable in the intended way |
| Input validation | Filter and validate all input at trust boundaries for expected formats, types, lengths, ranges |
| Output encoding | Properly escape and encode output destined for other systems or interpreters |
| Null sandboxing | Sandbox null values to input checks and database communication; use exceptions for exceptional cases |
| Error handling | Comprehensive and strategic handling of unexpected conditions |
| Immutable data structures | For threaded programming to prevent insecure modification, ensure thread safety, prevent race conditions |

These are verifiable items that AppSec can assess.

---

### S6.4.1 Canonical Input Handling

Input handling applies a minimal acceptable range for each parameter at the point of input through three key practices:

| Practice | Description |
| :--- | :--- |
| Canonicalization/Normalization | Ensures input conforms to expected formats, types, lengths, and ranges before processing |
| Sanitization | Cleans input to remove or neutralize potentially harmful content; prevents injection attacks |
| Validation | Checks input meets specific criteria; always prefer allowing explicit values over rejecting unexpected ones |

---

### S6.4.1.1 The Request Surface Minimization Principle

Avoid the assumption that the entire request is intended to be processed. Access **specific values** within the request rather than all values.

| Benefit | Description |
| :--- | :--- |
| Injection avoidance | Avoids certain types of injection attacks |
| Resilience | Ignores out-of-scope values, preserving stability |
| Fraud detection | System can analyze requests for deviation without affecting operation |
| Logging | Requests deviating from expectation should be logged or rejected in sensitive situations |

---

### S6.4.1.2 The Derived Integrity Principle

> Any value critical to the integrity of a system's state or business logic **must** be derived or calculated in a trusted context. It should never be accepted directly from a client.

**Analogy**: You would never let a customer walk into a store and tell the cashier how much an item costs. The price is non-negotiable; it's derived from the store's own trusted system.

| Candidate | Application |
| :--- | :--- |
| Pricing and Totals | Calculated server-side from product IDs and quantities, referencing a secure price database |
| User Permissions | Loaded from server-side session or database, never from client request |
| Object State | Managed by internal state machine, not dictated by client parameters |

This is a direct application of the SSEM attribute of **Integrity** (S3.2.3.2). The server accepts the client's *intent* but does not allow the client to violate the integrity of the operation.
