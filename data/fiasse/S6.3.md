# S6 Practical Guidance for Secure Software Development

## S6.3 The Flexibility Principle

A key concept from threat modeling is the identification of **trust boundaries** — points where data passes between entities with different levels of trust. Trust boundaries require heightened control over data and process execution.

Software engineers value flexibility in code. Attackers seek **uncontrolled flexibility** as a means to force the application to deviate from intended behavior.

> The issue is not flexibility itself, but the **exposure of flexibility through careless handling of trust boundaries**.

### The Turtle Analogy

Think of trust boundaries like the hard shell of a turtle:

| Part | Represents |
| :--- | :--- |
| Hard shell | Critical trust boundary points where external data is carefully controlled |
| Soft, flexible interior | The bulk of the application's logic |

Just as the shell protects the interior from attack, well-handled trust boundaries protect the core application from harmful external influences.

### Security Implications

| SSEM Attribute | How Control Supports It |
| :--- | :--- |
| Integrity | Control ensures trustworthy execution |
| Resilience | Controlled flexibility prevents unpredictable behavior |
| Maintainability | Flexibility is necessary for ongoing maintainability |

Defining and communicating trust boundaries during the design phase makes it clear which areas need tight control. Data Flow Diagrams are invaluable for identifying these boundaries. Strict input handling at trust boundary entry points allows the developer to focus on flexible code without worrying about unexpected input.
