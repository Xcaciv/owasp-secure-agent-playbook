# S5 Addressing Common AppSec Anti-Patterns

## S5.1 The "Shoveling Left" Phenomenon

"Shoveling Left" is defined as the practice of supplying impractical information to developers and leaving the responsibility on them to make sense of it. This anti-pattern undermines AppSec's credibility and leads to developer disengagement.

### S5.1.1 Ineffective Vulnerability Reporting

Dumping raw findings from security scanning tools directly into the development team's backlog lacks context, prioritization, and actionable guidance. While initial progress might be observed, momentum is typically lost and issues reappear (the "whack-a-mole" effect).

| Step | Better Practice |
| :--- | :--- |
| 1. Focus on True Positives | Validate findings to ensure accuracy |
| 2. Analyze Trends | Look for patterns across findings and tools to identify systemic issues |
| 3. Identify Root Causes | Address systemic issues rather than just symptoms |
| 4. Prioritize by Impact | Focus on issues that are plentiful or affect sensitive resources |
| 5. Collaborate on Solutions | Work with development to formulate wide-impact solutions instead of line-level mitigations |
| 6. Verify Fixes | Ensure remediation is effective and consider automated regression tests |

### S5.1.2 Pitfalls of Exploit-First Training

Security training that primarily emphasizes exploitation techniques ("learn the hack to stop the attack") is another manifestation of "Shoveling Left."

| Problem | Description |
| :--- | :--- |
| Mismatched skills | Does not equip developers with engineering principles for daily tasks |
| Missing SSEM context | Fails to teach how to identify or build code with inherently securable qualities |
| False sense of security | Superficial understanding of risks without practical implementation knowledge |
| Wrong focus | Should emphasize better design and implementation instead of line-level mitigations |
