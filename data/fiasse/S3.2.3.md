# S3 The Securable Principle

## S3.2 Core Securable Attributes

### S3.2.3 Reliability

**Definition**: "The degree to which a system, product or component performs specified functions under specified conditions for a specified period of time" (ISO/IEC 25010:2011).

In SSEM, this means the software operates consistently and predictably, even under adverse conditions or when facing unexpected inputs or attacks.

---

#### S3.2.3.1 Availability

**Definition**: "The property of a system or a system resource being accessible and usable upon demand by an authorized system entity, according to performance specifications for the system" (RFC 4949).

| Aspect | Security Implication |
| :--- | :--- |
| DDoS resilience | System designed to resist denial-of-service attacks |
| Quick recovery | System recovers quickly from failures or disruptions |
| Authorized access | Remains accessible to authorized users under adverse conditions |

---

#### S3.2.3.2 Integrity

**Definition**: Encompasses both data integrity ("the property that data has not been changed, destroyed, or lost in an unauthorized or accidental manner") and system integrity ("a system performs its intended function in an unimpaired manner free from unauthorized manipulation") (RFC 4949).

| Measure | Description |
| :--- | :--- |
| Cryptographic hashing | Prevent unauthorized modification of data |
| Checksums | Detect data corruption in transit or at rest |
| Access controls | Prevent unauthorized modification of data and system state |
| Holistic assurance | System trusted entirely, not just in isolated components |

Integrity ensures the system operates correctly and consistently, even in the face of potential threats or failures.

---

#### S3.2.3.3 Resilience

**Definition**: "The ability of a system to: (a) continue to operate during and after a failure of some part of the system (i.e., provide a degree of fault tolerance); and (b) recover from the failure and restore full operations" (RFC 4949).

Resilience inherently includes **fault tolerance**: "the ability of a system to continue to operate correctly even though a component has failed" (RFC 4949).

| Strategy | Description |
| :--- | :--- |
| Defensive coding | Writing code that anticipates input outside expected bounds and ignores those inputs rather than failing |
| Predictable execution | Ensuring code behaves consistently under various conditions |
| Strong trust boundaries | Clearly defining areas of the codebase that exert strictly controlled execution |
| Robust error handling | Managing partial failures effectively with recovery mechanisms |
