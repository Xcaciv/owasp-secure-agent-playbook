# S4 Integrating Security into Development Strategy

## S4.4 Early Integration: Planning and Requirements

FIASSE advocates for setting expectations at the earliest stages of development, particularly during planning and requirements definition. This ensures security is a foundational design aspect rather than an afterthought.

> Fixing vulnerabilities in the design phase costs **100x less** than addressing them in production [VulnCosts].

### Key Deliverables

| Deliverable | Description |
| :--- | :--- |
| Security Features | Specific security features that must be implemented (authentication mechanisms, encryption requirements, access controls) |
| Threat Scenarios | Potential misuse cases or attack paths relevant to the feature being developed |
| Security Acceptance Criteria | Specific, testable conditions that a feature must meet to be considered secure; enables QA to perform security testing |

By embedding security into foundational design decisions via requirements, attributes like Trustworthiness, Integrity, and Fault Tolerance are more effectively realized. This allows development teams to address security as part of their standard workflow.

Detailed methods for integrating security into requirements are discussed in S6.1.2.
